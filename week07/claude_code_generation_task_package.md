## 1. 背景说明

这是一个按周递进的 LLM 学习项目：

```text
LlmResearchProject/
├─ week03/  手写 HTTP API 的中文问答助手
├─ week04/  LangChain 模型封装与聊天助手
├─ week05/  结构化输出聊天机器人
├─ week06/  简单 Agent 原型
└─ week07/  本次需要完善的可演示 Agent
```

Week07 当前是 Week06 原型的复制版。它已有“读取本地资料、三段式回答、模型 fallback、命令行入口”等基础能力，但默认路径仍依赖模型服务，资料路径和文案仍指向 Week06，缺少清晰的检索依据展示、可安全复现的工具失败场景、持久化运行记录和 Week07 专属验收材料。

本次要实现的固定中文任务场景是：**大模型入门课程资料问答助手**。

它不是通用联网搜索 Agent。它只回答 `week07/local_knowledge.md` 中整理的课程资料，目标是在没有网络、没有 API Key、没有 Ollama/DeepSeek 服务时，也能稳定完成三种现场演示：

1. 直接回答：介绍自身能力和使用范围，不调用工具。
2. 调用工具：从本地课程资料中检索并回答，同时展示依据。
3. 工具失败：安全模拟资料读取失败，展示失败原因和降级结果。

运行时不得联网。用户提供的外部资料链接只用于编写本地知识摘要时的参考记录：

- https://zhuanlan.zhihu.com/p/2000239845751137469
- https://github.com/datawhalechina/llm-universe
- https://github.com/datawhalechina/llm-cookbook

不要复制外部来源的大段原文；应在本地 Markdown 中使用简短中文摘要，并保留参考链接说明。

### 3.2 `week07/local_knowledge.md`

### 3.3 `week07/simple_agent.py`

在不修改 Week03—Week06 的前提下，实现默认离线、确定性的 Week07 演示流程。

必须实现或改造出下列职责清晰的能力：

1. **输入路由**
   - 直接回答：问候、帮助、`你能帮助我做什么？`、使用方式等。
   - 本地检索：课程概念问题，或显式包含“根据资料”“知识库”“文档”“课程资料”等请求。
   - 范围外：既非帮助也非课程资料问题时，直接说明 Agent 的资料范围和限制。
   - 默认路由不得由 LLM 决定，必须可离线稳定运行。

2. **局部检索工具**
   - 读取 `week07/local_knowledge.md`。
   - 按 Markdown 二级标题切分成资料片段。
   - 使用可解释的中文关键词匹配排序：标题命中优先，正文命中次数其次。
   - 返回最相关的 1—2 个片段，并保留来源路径、章节标题、命中词和片段序号。
   - 无命中不等于读取失败：应返回“当前资料未覆盖该问题”。

3. **固定回答格式**

所有 `run_agent()` 返回结果都必须包含以下四段：

```text
结论：
<中文回答>

依据：
<资料来源、章节、检索方式、命中词；或未调用工具/失败原因>

是否调用工具：
<是 / 否 / 是，但失败>

运行记录：
<run_id 与日志写入位置，或日志失败原因>
```

工具成功时，依据必须可定位，例如：

```text
本地来源：week07/local_knowledge.md#提示词模板；
检索方式：本地关键词检索；命中词：提示词、模板。
```

4. **失败处理**
   - 增加 CLI 开关 `--simulate-tool-failure`。
   - 此开关只能对本次调用模拟工具读取失败，绝不能重命名、删除、改写真实资料文件。
   - 模拟或真实读取失败时，结论必须包含“未获取到资料”，依据展示可读错误，工具状态必须为“是，但失败”。
   - 任何文件读取、日志写入、模型导入或模型服务异常都不得让 CLI 输出 traceback 后崩溃。

5. **运行记录**
   - 每次调用生成唯一 `run_id`。
   - 将一行 JSON 追加到 `week07/runs/` 下的 JSONL 文件，使用 UTF-8。
   - 每条记录至少包含：时间、run_id、用户输入、路由、是否调用工具、工具状态、来源、章节、命中词、错误信息、回答摘要、日志写入状态。
   - 日志写入失败不得影响主回答；`运行记录` 段应如实说明失败原因。

6. **CLI 兼容性**
   - 保留现有单次问题调用形式。
   - 保留 `--no-stream`，使手工验收可以得到完整文本。
   - 新增 `--simulate-tool-failure`，并确保它不会被误当作用户问题文本。
   - 现有交互模式可以保留，但本次验收重点是单次命令行调用。

## 4. 输入输出要求

### 4.1 必须支持的输入

| 场景 | 命令或输入 | 预期路由 |
|---|---|---|
| 直接回答 | `你能帮助我做什么？` | 不调用工具 |
| 工具成功 | `根据课程资料，提示词模板由哪些部分组成？` | 本地检索 |
| 工具失败 | `--simulate-tool-failure "根据课程资料，什么是 RAG？"` | 读取失败降级 |
| 范围外 | `帮我查询今天上海的天气` | 直接说明不能联网，不读取资料 |
| 空输入 | `""` 或空白 | 输入提示，不调用工具 |

### 4.2 状态语义

| 情况 | `是否调用工具` |
|---|---|
| 问候、帮助、能力说明、范围外说明、空输入 | `否` |
| 已成功读取本地资料（包括资料无命中） | `是` |
| 尝试读取资料但失败或模拟失败 | `是，但失败` |

### 4.3 输出约束

- 输出语言为中文。
- 输出必须稳定：同一输入重复执行时，路由、状态和依据章节相同。
- 不得用“模型已调用”“资料已读取”等无法验证的说法。
- 工具成功必须有真实章节引用；工具失败不得带假引用。
- 不得直接展示敏感配置、API Key、异常堆栈或完整本地绝对路径。

## 5. 边界条件

- 文件不存在、无权限、编码/系统 I/O 错误：返回失败降级结果，不崩溃。
- 本地资料为空：作为“可读取但无有效资料”处理，不能伪造章节。
- 查询只包含标点、超长文本或特殊字符：不崩溃；可返回范围提示或资料未覆盖说明。
- 查询同时命中多个主题：最多返回相关度最高的 1—2 个章节，依据中逐一列出。
- 大小写不同的英文术语（如 `rag` / `RAG`）：路由和检索应尽量一致。
- 日志目录不存在：程序应创建；创建或写入失败时，主回答仍必须完整。
- LangChain 未安装、云端 Key 缺失、Ollama 未启动：默认三种演示路径仍必须成功，不应因此改变答案。
- `--simulate-tool-failure` 必须只影响当前进程，退出后真实资料仍可正常读取。

- 使用 Python 现有风格：4 空格缩进、UTF-8、中文用户提示、英文或中文均可的清晰函数名和 docstring。
